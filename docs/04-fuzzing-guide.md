# Гайд по фаззинг-тестированию для начинающего

## 1. Что такое фаззинг простыми словами

Фаззинг — это способ проверки системы, при котором мы подаем на вход много неожиданного, некорректного или случайного ввода и смотрим:

- не падает ли приложение;
- корректно ли оно отклоняет мусорные данные;
- не пропускает ли опасные значения.

Для нашего проекта фаззинг особенно полезен в местах:

- входа в систему;
- создания пользователей;
- проверки ролей;
- создания тестов, вопросов и вариантов ответов;
- отправки ответов студентом.

## 2. Что именно нужно проверять в этой системе

### Для пользователей

- пустой логин;
- слишком длинный логин;
- запрещенные символы;
- несуществующая роль;
- пустой пароль;
- очень длинный пароль;
- отключенный пользователь.

### Для тестов

- пустое название;
- тест без вопросов;
- вопрос без вариантов;
- только один вариант ответа;
- несколько правильных ответов в вопросе, если модель поддерживает только один;
- очень длинные строки;
- неверные идентификаторы.

### Для попыток прохождения

- ответ на несуществующий вопрос;
- выбор варианта из другого вопроса;
- повторяющиеся ответы;
- попытка пройти неопубликованный тест;
- попытка отправить пустой набор ответов.

В проекте подготовлены автоматические тесты:

- [tests/test_api.py](../tests/test_api.py)
- [tests/test_fuzzing.py](../tests/test_fuzzing.py)

Первый файл проверяет основные сценарии CRUD и ролей.  
Второй — подает набор «ломающих» или пограничных данных.

## 4. Как запустить проверки локально

Из корня проекта:

```bash
pytest
```

Если нужно запустить только фаззинг-набор:

```bash
pytest tests/test_fuzzing.py
```

## 5. Как интерпретировать результат

Хороший результат для фаззинга в этом проекте выглядит так:

- сервер не падает;
- на неправильный ввод возвращаются коды `400`, `401`, `403`, `404` или `422`;
- база данных не портится;
- роль пользователя не повышается из-за подмены запроса;
- студент не может получить доступ к функциям преподавателя или администратора.

Плохой результат:

- сервер падает с `500 Internal Server Error`;
- создается пользователь с ролью вне списка;
- можно опубликовать сломанный тест;
- студент может удалить тест или изменить пользователя;
- приложение зависает на необычном вводе.

## 6. Примеры ручного фаззинга через API

### Пример 1. Неверная роль

Отправить запрос на создание пользователя с ролью `superuser`.

Ожидание:

- сервер должен отклонить запрос;
- пользователь не должен появиться в БД.

### Пример 2. Сломанная структура теста

Отправить тест:

- без вопросов;
- с вопросом без правильного ответа;
- с вопросом, где все ответы правильные.

Ожидание:

- сервер должен вернуть ошибку валидации.

### Пример 3. Подмена прав

Войти как студент и попытаться:

- создать тест;
- удалить пользователя;
- посмотреть список всех пользователей.

Ожидание:

- сервер должен вернуть `403 Forbidden`.

## 7. Минимальный сценарий защиты на курсовой

Если преподаватель спросит, как проводилось фаззинг-тестирование, можно отвечать так:

1. Были выделены критические точки ввода: авторизация, роли, создание тестов и отправка ответов.
2. Для них были подготовлены негативные и пограничные наборы данных.
3. Проверка выполнялась автоматическими тестами `pytest`.
4. Основной критерий успеха — отсутствие аварий сервера и корректная обработка некорректного ввода.

## 8. Вывод

Даже для небольшой учебной системы фаззинг полезен, потому что быстро показывает, насколько надежно сервер обрабатывает мусорный ввод, нарушения ролевой модели и поврежденные структуры данных.
