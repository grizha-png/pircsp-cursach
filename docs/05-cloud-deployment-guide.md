# Облачное развертывание системы

## 1. Зачем вообще изучать облачное развертывание

После локальной реализации системы следующий логичный этап — сделать так, чтобы приложение было доступно из интернета:

- преподаватель и студент могли открыть его из браузера;
- не требовалось вручную переносить файлы между устройствами;
- проект было удобно показать на защите.

Для этого используется облачное развертывание.

## 2. Что именно нужно развернуть в нашем проекте

Для проекта «Интерактивный конструктор образовательных тестов» нужно развернуть:

- frontend;
- backend API;
- хранилище данных.

В учебной версии проекта backend работает с SQLite. Это удобно локально, но в облаке нужно помнить:

- часть платформ использует **эфемерную файловую систему**;
- при перезапуске контейнера файл БД может потеряться;
- для устойчивого облачного размещения лучше использовать persistent disk или перейти на PostgreSQL.

## 3. Актуальные примеры облаков с бесплатным или условно бесплатным стартом

Ниже перечислены варианты, которые были актуальны **по состоянию на 27 мая 2026 года**.

### 3.1 Render

Подходит для:

- backend на Python;
- frontend как static site;
- Docker-based деплоя.

Что полезно для нас:

- у Render есть бесплатный Hobby workspace;
- можно бесплатно развертывать некоторые web services и static sites;
- подходит для демонстрационных и учебных проектов.

Ограничения:

- возможны cold start;
- бесплатный режим не предназначен для production-нагрузки;
- нужно внимательно проверить политику persistent storage для выбранного типа сервиса.

Источник: Render Docs, `Deploy for Free`  
https://render.com/free

### 3.2 Railway

Подходит для:

- небольших fullstack-приложений;
- быстрого деплоя из GitHub;
- экспериментов с Docker и сервисами.

Что полезно для нас:

- Railway предоставляет free trial;
- после trial действует Free plan с ограниченным бесплатным кредитом в месяц.

Ограничения:

- часть trial-функций зависит от верификации аккаунта;
- лимиты по ресурсам и сети нужно проверять перед деплоем;
- для SQLite нужно отдельно продумать вопрос постоянного хранения.

Источник: Railway Docs, `Free Trial` и `Pricing Plans`  
https://docs.railway.com/pricing/free-trial  
https://docs.railway.com/pricing/plans

### 3.3 Vercel

Подходит прежде всего для:

- фронтенда;
- небольших personal-проектов;
- статических сайтов и frontend-first решений.

Что полезно для нас:

- план Hobby бесплатный;
- удобно размещать клиентскую часть отдельно от backend;
- можно использовать для публикации интерфейса проекта.

Ограничения:

- backend-часть на постоянном серверном процессе разворачивать не так удобно, как на Render/Railway;
- для нашей архитектуры Vercel логичнее использовать под frontend.

Источник: Vercel Docs, `Hobby Plan`  
https://vercel.com/docs/accounts/plans/hobby

### 3.4 AWS Free Tier

Подходит для:

- знакомства с IaaS;
- развертывания виртуальной машины;
- ручного управления стеком.

Что полезно для нас:

- можно получить бесплатные ресурсы на ограниченный период;
- можно развернуть backend, frontend и reverse proxy на одной VM;
- это хороший учебный вариант для понимания «настоящего» серверного окружения.

Ограничения:

- нужно внимательно следить за лимитами;
- существует риск выйти за пределы бесплатного уровня;
- настройка сложнее, чем на PaaS-платформах.

Источники: AWS Free Tier и документация EC2  
https://aws.amazon.com/free/  
https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-free-tier-usage.html

### 3.5 Google Cloud Free Tier / Free Trial

Подходит для:

- учебных proof-of-concept;
- размещения приложений на виртуальных машинах и managed-сервисах;
- знакомства с облачной экосистемой Google.

Что полезно для нас:

- новым пользователям дается стартовый кредит;
- часть продуктов имеет free tier;
- можно развернуть тестовый backend и статический frontend.

Ограничения:

- нужно контролировать расход кредита;
- конфигурация сложнее, чем у специализированных PaaS.

Источники: Google Cloud Free Program  
https://cloud.google.com/free  
https://cloud.google.com/free/docs/compute-getting-started

## 4. Что выбрать именно для этого проекта

Для данной курсовой работы практичнее всего такие варианты:

### Вариант A. Самый простой для демонстрации

- frontend на Vercel;
- backend на Render;
- для БД либо persistent disk, либо перенос на PostgreSQL.

### Вариант B. Самый понятный для единого деплоя

- один Dockerized backend/frontend стек на Render или Railway.

### Вариант C. Самый «инфраструктурный»

- одна виртуальная машина AWS или Google Cloud;
- запуск `docker compose up -d`;
- полный контроль над окружением.

## 5. Рекомендация по защите курсовой

Если задача — быстро показать работающий проект, лучше выбирать **Render или Railway**.

Если задача — показать понимание инфраструктуры глубже, можно дополнительно рассказать про вариант с **AWS EC2** или **Google Compute Engine**.

## 6. Что нужно изменить перед реальным облачным запуском

Перед публичным размещением проекта желательно:

- заменить SQLite на PostgreSQL;
- убрать демонстрационные пароли;
- хранить секреты только в переменных окружения;
- включить HTTPS;
- ограничить CORS;
- добавить резервное копирование БД;
- настроить логи и мониторинг.

## 7. Вывод

Облачное развертывание для небольшой учебной системы реально выполнить даже на бесплатных или условно бесплатных тарифах, но нужно внимательно смотреть на лимиты, cold start и постоянство хранилища. Для данного проекта наиболее реалистичный стартовый путь — Docker-деплой на Render или Railway, а для более устойчивой эксплуатации — переход на PostgreSQL и managed cloud-схему.
