# Выбор клиент-серверной архитектуры

## 1. Почему для проекта выбрана клиент-серверная архитектура

Для темы «Интерактивный конструктор образовательных тестов» оптимальна именно **клиент-серверная архитектура**, потому что:

- несколько пользователей работают с одними и теми же данными;
- требуется централизованная авторизация и проверка ролей;
- результаты прохождения должны храниться в единой точке;
- преподаватель и студент взаимодействуют с разными сценариями в одном приложении;
- систему необходимо контейнеризировать и подготовить к облачному развертыванию.

Локальное настольное приложение хуже подходит, так как усложняет совместный доступ и централизованное хранение данных. Полностью серверная HTML-система тоже возможна, но в учебном fullstack-проекте выгоднее явно разделить клиент и API.

## 2. Рассмотренные варианты

### Вариант A. Монолитное настольное приложение

Плюсы:

- просто начать разработку;
- нет необходимости в отдельном веб-клиенте.

Минусы:

- слабая поддержка многопользовательского режима;
- неудобное развертывание;
- хуже подходит под облачный сценарий;
- слабо соответствует заданию на fullstack CRUD и Docker.

### Вариант B. Веб-приложение с серверной генерацией страниц

Плюсы:

- проще единое приложение;
- можно быстро сделать формы и страницы.

Минусы:

- сложнее явно показать клиентскую и серверную части как отдельные артефакты;
- UI становится менее гибким для интерактивного конструктора вопросов.

### Вариант C. Клиент-серверное веб-приложение с REST API

Плюсы:

- естественное разделение на frontend и backend;
- хорошо соответствует требованиям курсовой;
- удобно контейнеризировать;
- удобно защищать архитектурно;
- легко расширять новыми сценариями и клиентами.

Минусы:

- чуть больше объем начальной настройки;
- необходимо продумать API и ролевую защиту.

## 3. Итоговый выбор

Выбран **вариант C**: веб-клиент + серверное REST API + база данных.

Архитектурный стиль решения:

- **клиент** — отдельное статическое веб-приложение;
- **сервер** — FastAPI-приложение, реализующее бизнес-логику и авторизацию;
- **хранилище** — SQLite как компактная база данных для учебного проекта.

## 4. Детализация клиент-серверной части

### 3.1.1 Структура клиент-серверной части

При проектировании приложения выбран подход с явным разделением ответственности между клиентской частью, серверной логикой и хранением данных. Для данного проекта использование микросервисной архитектуры или сложной гексагональной схемы не требуется, поскольку система имеет учебный характер и решает ограниченный набор задач: регистрацию и вход пользователей, управление пользователями, создание тестов, добавление вопросов и вариантов ответов, публикацию тестов, прохождение тестирования и сохранение результатов.

Клиентская часть реализована как статическое веб-приложение и размещена в каталоге `frontend/`. В текущей реализации она состоит из трех основных файлов:

- `index.html` — структура интерфейса и основные пользовательские блоки;
- `styles.css` — оформление страниц, панелей, карточек и форм;
- `app.js` — логика интерфейса, хранение состояния, отправка HTTP-запросов к API и обработка ответов сервера.

Клиент работает в браузере и не содержит собственной бизнес-логики предметной области. Его задача заключается в отображении интерфейса, сборе данных из форм, отправке запросов к серверу, показе результатов и переключении доступных разделов в зависимости от роли пользователя. В интерфейсе выделяются следующие основные части:

- блок входа в систему;
- блок регистрации студента;
- панель активной сессии пользователя;
- административный раздел для CRUD-управления пользователями;
- раздел преподавателя для создания, редактирования, публикации и удаления тестов;
- раздел преподавателя для просмотра попыток прохождения по выбранному тесту;
- раздел студента со списком опубликованных тестов;
- форма прохождения теста;
- блок истории собственных результатов студента.

Для взаимодействия с сервером используется единый клиентский слой обращения к API, реализованный в `app.js`. Он формирует HTTP-запросы, автоматически добавляет токен авторизации к защищенным операциям, обрабатывает ответы сервера, выводит сообщения об ошибках и обновляет данные на странице без полной перезагрузки интерфейса.

Серверная часть размещена в каталоге `src/app/` и реализована на `FastAPI`. Она построена по упрощенной многослойной схеме, которая хорошо подходит для учебного проекта и позволяет явно проследить путь запроса от клиента до базы данных. В серверной части выделены следующие уровни:

- `main.py` — точка сборки приложения, подключение `CORS`, инициализация БД, подключение маршрутов и тестовых данных;
- `core/` — конфигурация приложения и функции безопасности, включая загрузку `.env`, генерацию токенов и хэширование паролей;
- `api/` — общие зависимости FastAPI, в том числе открытие соединения с базой данных;
- `modules/` — предметные модули приложения;
- `validation/` — модели `Pydantic`, проверяющие корректность входных данных;
- `db/` — схема базы данных и функции доступа к данным;
- `seed.py` — начальное заполнение системы демонстрационными пользователями, тестами и попытками.

Предметные модули серверной части организованы по папкам:

- `modules/auth` — регистрация студента, вход, выход, получение текущего пользователя и проверки ролей;
- `modules/users` — просмотр, создание, изменение и удаление пользователей администратором;
- `modules/tests` — создание, редактирование, публикация, просмотр и удаление тестов;
- `modules/attempts` — отправка ответов студентом, подсчет результата и просмотр попыток.

Внутри модулей разделены маршруты API и бизнес-логика: файлы `router.py` принимают HTTP-запросы, а файлы `service.py` выполняют прикладные операции. Такой подход позволяет не смешивать обработку веб-запросов с правилами предметной области.

Таким образом, полный путь обработки запроса выглядит следующим образом: браузер отправляет HTTP-запрос к REST API, маршрут FastAPI принимает запрос, модели `Pydantic` выполняют валидацию входных данных, зависимости проверяют токен и роль пользователя, сервисный слой реализует прикладную логику, слой `repositories` выполняет SQL-операции, после чего результат возвращается клиенту в формате JSON.

### 3.1.2 Модель данных и ER-диаграмма

Модель данных отражает ключевые сущности предметной области и связи между ними. В текущей реализации схемы БД выделены следующие основные таблицы:

- `User` — учетная запись пользователя системы: логин, полное имя, хэш пароля, роль, признак активности и дата создания;
- `Session` — активная пользовательская сессия: токен, срок действия и дата создания;
- `Test` — тест, созданный преподавателем или администратором: владелец, название, описание, признак публикации и даты создания и обновления;
- `Question` — вопрос внутри теста: формулировка, пояснение и позиция в тесте;
- `Option` — вариант ответа на вопрос: текст варианта, признак правильного ответа и позиция;
- `Attempt` — попытка прохождения теста студентом: ссылка на тест и пользователя, набранный балл, количество вопросов и дата отправки;
- `AttemptAnswer` — конкретный ответ в рамках попытки: выбранный вариант, вопрос и признак корректности ответа.

Между таблицами установлены следующие основные связи:

- связь «один-ко-многим» между `User` и `Session`, поскольку один пользователь может иметь несколько активных или исторических сессий;
- связь «один-ко-многим» между `User` и `Test`, поскольку один преподаватель может создать несколько тестов;
- связь «один-ко-многим» между `Test` и `Question`, поскольку тест содержит набор вопросов;
- связь «один-ко-многим» между `Question` и `Option`, поскольку каждый вопрос содержит несколько вариантов ответа;
- связь «один-ко-многим» между `User` и `Attempt`, поскольку студент может проходить разные тесты;
- связь «один-ко-многим» между `Test` и `Attempt`, поскольку один тест может быть пройден многими студентами;
- связь «один-ко-многим» между `Attempt` и `AttemptAnswer`, поскольку одна попытка состоит из набора ответов;
- связь «один-ко-многим» между `Question` и `AttemptAnswer`, поскольку в ответах фиксируется ссылка на конкретный вопрос;
- связь «один-ко-многим» между `Option` и `AttemptAnswer`, поскольку в ответах хранится выбранный вариант.

С точки зрения предметной области важна также производная связь «многие-ко-многим» между студентами и тестами, которая реализуется через таблицу `Attempt`: один студент может проходить много тестов, а один тест может иметь много прохождений. Фиксация отдельных ответов вынесена в таблицу `AttemptAnswer`, что позволяет сохранять структуру прохождения и использовать ее для последующего просмотра результатов.

Во всех ключевых связях используются внешние ключи. Для связанных сущностей настроены каскадные правила удаления, что обеспечивает целостность данных при удалении пользователя, теста, вопроса или попытки. Например, удаление теста автоматически приводит к удалению его вопросов, вариантов ответов, попыток и ответов в рамках этих попыток.

ER-диаграмма базы данных представлена в файле [er-diagram.uml](./uml/er-diagram.uml).

### 3.1.3 UML-диаграммы архитектуры

Для формализованного описания архитектуры подготовлен набор UML-диаграмм, отражающих как статический, так и динамический аспекты системы. Использованы следующие диаграммы:

- диаграмма прецедентов — отражает основные сценарии использования системы для гостя, студента, преподавателя и администратора. Среди ключевых прецедентов выделены регистрация и вход, управление пользователями, создание и редактирование тестов, публикация тестов, прохождение тестирования и просмотр результатов;
- диаграмма классов — описывает основные сущности предметной области и их связи. На ней представлены сущности `User`, `Session`, `Test`, `Question`, `Option`, `Attempt`, `AttemptAnswer`, а также перечисление ролей пользователей. Связи между сущностями соответствуют структуре базы данных и поддерживают основные операции бизнес-логики;
- диаграмма последовательностей — иллюстрирует динамический аспект системы на примере ключевого сценария прохождения теста студентом. На ней показано взаимодействие клиента с REST API, проверка токена сессии и роли пользователя, валидация входных данных, обращение к сервисному слою, чтение данных теста, проверка ответов и сохранение результата в базе данных;
- диаграмма компонентов и развертывания — объединяет логическую структуру решения и схему размещения приложения в контейнеризированном окружении. На ней показаны браузерный клиент, контейнер `frontend` на базе `nginx`, контейнер `backend` с `FastAPI`, внутренние слои серверной части и подключенный том с файлом базы данных SQLite.

Таким образом, UML-диаграммы позволяют формально зафиксировать выбранную архитектуру, состав сущностей, взаимодействие компонентов и порядок выполнения ключевых пользовательских сценариев.

## 6. Причины выбора FastAPI

FastAPI подходит для проекта по следующим причинам:

- хорошо решает задачу построения REST API;
- поддерживает валидацию моделей из коробки;
- дает автоматическую документацию API;
- удобен для демонстрации CRUD и ролевого доступа;
- достаточно легковесен для учебного проекта.

## 7. Причины выбора SQLite

SQLite выбран осознанно как учебное хранилище:

- не требует отдельного сервера БД;
- легко запускается локально и в контейнере;
- достаточно для небольшой демонстрационной системы;
- упрощает воспроизводимость проекта преподавателем.

Ограничение: для реального промышленного масштабирования лучше перейти на PostgreSQL.

## 8. Безопасность и контроль доступа

В архитектуре предусмотрены следующие меры:

- пароль хранится не в открытом виде, а в виде хэша;
- сервер выдает токен-сессию после успешного входа;
- все защищенные запросы проверяют токен;
- роли проверяются на стороне сервера;
- недопустимые значения ролей и некорректные структуры тестов отклоняются валидаторами.

## 9. Соответствие методологии Twelve-Factor App

В рамках учебного проекта частично учтены принципы `12-factor`:

- конфигурация выносится в переменные окружения;
- код и конфигурация разделены;
- приложение готово к запуску в контейнере;
- зависимости backend описываются отдельно;
- сервисы рассматриваются как подключаемые ресурсы;
- проект подготовлен к облачному запуску.

Для учебной SQLite-реализации некоторые ограничения остаются, но сама структура уже ориентирована на последующий перенос в более «облачное» окружение.

## 11. Вывод

Клиент-серверная архитектура выбрана потому, что она наилучшим образом решает реальные задачи предметной области: многопользовательскую работу, единое хранение данных, разграничение ролей, централизованную проверку прав и подготовку к облачному развертыванию.

Для курсового проекта такой вариант также наиболее удачен с учебной точки зрения: он позволяет показать и анализ, и проектирование, и полноценную реализацию fullstack CRUD-системы.
