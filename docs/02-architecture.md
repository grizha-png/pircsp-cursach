# Выбор клиент-серверной архитектуры

## 1. Почему для проекта выбрана клиент-серверная архитектура

Для темы «Интерактивный конструктор образовательных тестов» оптимальна именно **клиент-серверная архитектура**, потому что:

- несколько пользователей работают с одними и теми же данными;
- требуется централизованная авторизация и проверка ролей;
- результаты прохождения должны храниться в единой точке;
- преподаватель и студент взаимодействуют с разными сценариями в одном приложении;
- систему необходимо контейнеризировать и подготовить к облачному развертыванию.

Локальное настольное приложение хуже подходит, так как усложняет совместный доступ и централизованное хранение данных. Полностью серверная HTML-система тоже возможна, но в учебном fullstack-проекте выгоднее явно разделить клиент и API.

## 2. Рассмотренные варианты

### Вариант A. Монолитное настольное приложение

Плюсы:

- просто начать разработку;
- нет необходимости в отдельном веб-клиенте.

Минусы:

- слабая поддержка многопользовательского режима;
- неудобное развертывание;
- хуже подходит под облачный сценарий;
- слабо соответствует заданию на fullstack CRUD и Docker.

### Вариант B. Веб-приложение с серверной генерацией страниц

Плюсы:

- проще единое приложение;
- можно быстро сделать формы и страницы.

Минусы:

- сложнее явно показать клиентскую и серверную части как отдельные артефакты;
- UI становится менее гибким для интерактивного конструктора вопросов.

### Вариант C. Клиент-серверное веб-приложение с REST API

Плюсы:

- естественное разделение на frontend и backend;
- хорошо соответствует требованиям курсовой;
- удобно контейнеризировать;
- удобно защищать архитектурно;
- легко расширять новыми сценариями и клиентами.

Минусы:

- чуть больше объем начальной настройки;
- необходимо продумать API и ролевую защиту.

## 3. Итоговый выбор

Выбран **вариант C**: веб-клиент + серверное REST API + база данных.

Архитектурный стиль решения:

- **клиент** — отдельное статическое веб-приложение;
- **сервер** — FastAPI-приложение, реализующее бизнес-логику и авторизацию;
- **хранилище** — SQLite как компактная база данных для учебного проекта.

## 4. Логическая схема решения

Пользователь работает через браузер. Браузер открывает клиентское приложение, которое:

- отображает формы входа;
- запрашивает и отправляет данные по HTTP;
- показывает разные экраны в зависимости от роли.

Серверное приложение:

- принимает HTTP-запросы;
- аутентифицирует пользователя по токену;
- проверяет роль и права доступа;
- валидирует входные данные;
- выполняет CRUD-операции;
- сохраняет и читает данные из БД;
- возвращает JSON-ответы клиенту.

База данных хранит:

- пользователей;
- сессии;
- тесты;
- вопросы;
- варианты ответов;
- попытки;
- ответы в рамках попыток.

## 5. Основные архитектурные компоненты

### 5.1 Клиентский слой

Функции клиентского слоя:

- форма входа;
- панель администратора для управления пользователями;
- панель преподавателя для конструирования тестов;
- панель студента для прохождения тестов;
- отображение результатов и ошибок валидации.

### 5.2 Серверный слой

Функции серверного слоя:

- REST API;
- бизнес-правила;
- валидация входных моделей;
- контроль ролей и доступа;
- подсчет результатов тестов;
- выдача тестовых данных при первом запуске.

### 5.3 Слой данных

Функции слоя данных:

- хранение сущностей предметной области;
- обеспечение целостности связей;
- хранение активных сессий;
- хранение истории попыток.

## 6. Причины выбора FastAPI

FastAPI подходит для проекта по следующим причинам:

- хорошо решает задачу построения REST API;
- поддерживает валидацию моделей из коробки;
- дает автоматическую документацию API;
- удобен для демонстрации CRUD и ролевого доступа;
- достаточно легковесен для учебного проекта.

## 7. Причины выбора SQLite

SQLite выбран осознанно как учебное хранилище:

- не требует отдельного сервера БД;
- легко запускается локально и в контейнере;
- достаточно для небольшой демонстрационной системы;
- упрощает воспроизводимость проекта преподавателем.

Ограничение: для реального промышленного масштабирования лучше перейти на PostgreSQL.

## 8. Безопасность и контроль доступа

В архитектуре предусмотрены следующие меры:

- пароль хранится не в открытом виде, а в виде хэша;
- сервер выдает токен-сессию после успешного входа;
- все защищенные запросы проверяют токен;
- роли проверяются на стороне сервера;
- недопустимые значения ролей и некорректные структуры тестов отклоняются валидаторами.

## 9. Соответствие методологии Twelve-Factor App

В рамках учебного проекта частично учтены принципы `12-factor`:

- конфигурация выносится в переменные окружения;
- код и конфигурация разделены;
- приложение готово к запуску в контейнере;
- зависимости backend описываются отдельно;
- сервисы рассматриваются как подключаемые ресурсы;
- проект подготовлен к облачному запуску.

Для учебной SQLite-реализации некоторые ограничения остаются, но сама структура уже ориентирована на последующий перенос в более «облачное» окружение.

## 11. Вывод

Клиент-серверная архитектура выбрана потому, что она наилучшим образом решает реальные задачи предметной области: многопользовательскую работу, единое хранение данных, разграничение ролей, централизованную проверку прав и подготовку к облачному развертыванию.

Для курсового проекта такой вариант также наиболее удачен с учебной точки зрения: он позволяет показать и анализ, и проектирование, и полноценную реализацию fullstack CRUD-системы.
